# AI 工单项目学习问答：AI 操作审计日志、Dashboard 与部署工程化

> 适用项目：AI 工单管理系统  
> 适用阶段：工单详情闭环、AI 操作审计日志、管理员 Dashboard、Docker 部署优化  
> 写法说明：每个问题都提供两种解释：
>
> - **专业术语版**：适合写到项目文档、简历、答辩、面试中。
> - **初学者理解版**：用更直白的话帮助你真正理解。

# 6.4 AI 操作审计日志

## 一、基础问题

### 1. 什么是审计日志？

**专业术语版：**  
审计日志（Audit Log）是系统对关键业务操作、权限行为、数据变更和外部服务调用进行持久化记录的一类安全日志。它强调可追溯性、不可抵赖性和合规性，通常用于问题排查、安全审计、责任定位和风控分析。

在当前 AI 工单系统中，审计日志需要记录：谁触发了 AI、AI 判断了什么意图、准备调用什么工具、是否进入 pending_action、用户是否确认、最终执行是否成功。

**初学者理解版：**  
审计日志就是系统的“操作记录本”。  
比如 staff 让 AI 把 3 号工单改成已完成，系统不能只真的改了状态，还要记下来：是谁让 AI 改的、AI 准备改什么、有没有确认、最后有没有成功。

---

### 2. 审计日志和普通运行日志有什么区别？

**专业术语版：**  
普通运行日志（Runtime Log）主要服务于系统调试和运行状态观测，例如异常堆栈、接口耗时、SQL 执行情况。审计日志主要服务于业务行为追踪和安全合规，关注“主体、动作、对象、结果、时间、上下文”。

普通日志更偏技术运行状态；审计日志更偏业务操作证据。

**初学者理解版：**  
普通日志像程序员看的“报错信息”；审计日志像管理员看的“谁做了什么”。  
例如 Java 报了一个 NullPointerException，这是普通运行日志；staff 用 AI 修改了工单状态，这是审计日志。

---

### 3. 为什么 AI Agent 必须记录 tool call？

**专业术语版：**  
AI Agent 的 tool call 代表模型从自然语言推理转向系统行为执行的关键边界。记录 tool call 可以追踪模型意图、工具选择、参数构造、权限判断和执行结果，避免 AI 黑箱执行敏感操作。

在你的系统中，查询工单、创建工单、修改状态、保存 AI 回复建议都属于 tool call 相关行为，尤其写操作必须记录。

**初学者理解版：**  
AI 不能只说“我帮你处理了”，系统必须知道它到底调用了哪个工具。  
比如用户说“把工单 3 改成已完成”，AI 可能调用 `update_ticket_status`，这个调用必须记下来，否则后面出了问题不知道是谁改的、怎么改的。

---

### 4. 为什么失败操作也要记录？

**专业术语版：**  
失败操作是审计链路的重要组成部分。它能反映系统异常、权限拦截、参数错误、外部服务失败和潜在攻击尝试。只记录成功会造成审计盲区，无法还原完整操作过程。

**初学者理解版：**  
失败也很重要。  
比如用户没有权限却一直尝试让 AI 修改别人的工单，如果不记录失败操作，管理员就看不到这种异常行为。

---

### 5. 为什么取消操作也要记录？

**专业术语版：**  
取消操作表示用户中止了 pending_action，是 Human-in-the-loop 流程中的显式决策。记录取消行为可以证明系统没有执行该写操作，也能帮助分析 AI 建议是否经常被用户拒绝。

**初学者理解版：**  
用户点了“取消”，这也是一次重要决定。  
比如 AI 建议创建一个工单，但 staff 取消了，系统要能证明这次没有真的创建。

---

### 6. 为什么权限不足也要记录？

**专业术语版：**  
权限不足属于安全审计事件。记录 FORBIDDEN 行为可以帮助发现越权访问、角色配置错误、恶意尝试和前后端权限不一致问题。尤其在 AI Agent 场景中，权限失败能证明 Agent 没有绕过后端安全边界。

**初学者理解版：**  
有人想做自己不能做的事，也要记下来。  
比如普通 USER 让 AI 修改别人的工单，后端拒绝了，这个拒绝也要进日志。

---

### 7. 什么是 conversationId？

**专业术语版：**  
conversationId 是一次 AI 会话或一次连续交互流程的上下文标识，用于关联用户输入、AI 意图识别、工具调用、pending_action、确认、取消和最终执行结果。

它不是用户 ID，而是“这轮 AI 对话链路”的 ID。

**初学者理解版：**  
conversationId 就像一次 AI 对话的“订单号”。  
用户和 AI 多轮交流时，系统用它把这些消息串起来，知道哪些操作属于同一次对话。

---

### 8. conversationId 和 userId 有什么区别？

**专业术语版：**  
userId 表示操作主体，即哪个用户发起操作；conversationId 表示交互上下文，即该操作属于哪一次 AI 会话。一个 userId 可以对应多个 conversationId，一个 conversationId 通常属于某个用户的一次会话链路。

**初学者理解版：**  
userId 是“谁”；conversationId 是“哪一轮对话”。  
同一个用户今天可能问 AI 很多次，每一次对话都应该有不同的 conversationId。

---

### 9. 一次 AI 操作链路应该包含哪些日志？

**专业术语版：**  
完整 AI 操作链路通常包含：用户请求日志、意图识别日志、工具调用准备日志、pending_action 创建日志、用户确认或取消日志、执行结果日志、失败或权限异常日志。

在你的项目中，一次修改工单状态的链路可以是：

1. AI Controller 收到用户输入；
2. Python 返回 `UPDATE_TICKET_STATUS`；
3. Java 创建 `AI_UPDATE_STATUS_PENDING`；
4. 用户确认；
5. Java 执行状态修改；
6. 记录 `AI_UPDATE_STATUS_CONFIRMED`。

**初学者理解版：**  
一次 AI 操作不能只记最后结果，要把中间过程也记下来。  
例如“修改工单状态”要记录：用户说了什么、AI 理解成什么、有没有让用户确认、用户确认还是取消、最后有没有真的改成功。

---

### 10. 如何通过 conversationId 查完整链路？

**专业术语版：**  
在日志查询接口中支持 conversationId 过滤，并按 createdAt 升序排序，即可还原一次 AI 会话从请求到执行完成的完整链路。前端日志页可以提供 conversationId 搜索框，后端在 operation_log 中存储 conversationId 或将其放入 extraJson。

**初学者理解版：**  
在日志页输入同一个 conversationId，就能看到这次 AI 对话相关的所有记录。  
就像快递单号一样，用一个编号查到整个过程。

---

## 二、字段设计问题

### 11. operationSource 应该有哪些值？

**专业术语版：**  
operationSource 用于标识操作来源。建议至少包含：

- `MANUAL`：用户或工作人员在页面上直接操作；
- `AI`：由 AI Agent 触发、建议或驱动的操作。

如果后续扩展，可以增加 `SYSTEM` 表示系统定时任务或自动任务。

**初学者理解版：**  
operationSource 就是说明“这个操作是谁发起的方式”。  
是人直接点按钮做的，还是 AI 帮忙触发的。

---

### 12. actionType 应该怎么命名？

**专业术语版：**  
actionType 应采用稳定、可枚举、语义明确的命名方式，推荐格式为：

`来源_业务动作_阶段`

例如：

- `AI_QUERY_TICKET`
- `AI_CREATE_TICKET_PENDING`
- `AI_CREATE_TICKET_CONFIRMED`
- `AI_UPDATE_STATUS_CANCELLED`
- `AI_REPLY_SUGGESTION`
- `AI_REPLY_CONFIRMED`

命名要避免随意字符串，最好使用枚举或常量统一管理。

**初学者理解版：**  
actionType 就是“具体做了什么”。  
不要一会儿写 `reply_ai`，一会儿写 `AI_REPLY`，命名要统一，不然后面筛选日志会很乱。

---

### 13. targetType 是什么？

**专业术语版：**  
targetType 表示操作对象类型，用于说明本次操作针对的是哪一类资源。常见值包括：

- `TICKET`
- `REPLY`
- `USER`
- `AI_AGENT`
- `OPERATION_LOG`

在你的系统中，大部分 AI 操作的 targetType 都是 `TICKET` 或 `REPLY`。

**初学者理解版：**  
targetType 就是“操作的是哪种东西”。  
比如修改工单，targetType 就是 TICKET；保存回复，targetType 可以是 REPLY。

---

### 14. targetId 是什么？

**专业术语版：**  
targetId 表示具体操作对象的唯一标识，通常对应数据库主键。它和 targetType 组合后可以定位具体资源，例如 `TICKET#3`、`REPLY#12`。

**初学者理解版：**  
targetId 就是“具体哪个对象”。  
targetType 是 TICKET，targetId 是 3，就表示这次操作针对 3 号工单。

---

### 15. resultStatus 应该有哪些值？

**专业术语版：**  
resultStatus 用于表示操作结果状态，建议包括：

- `SUCCESS`：操作成功；
- `FAILED`：系统错误、参数错误或外部服务失败；
- `CANCELLED`：用户取消 pending_action；
- `FORBIDDEN`：权限不足；
- 可选：`PENDING`，表示操作进入待确认状态。

如果 pending 行为单独用 actionType 表示，例如 `AI_CREATE_TICKET_PENDING`，resultStatus 可以仍然是 `SUCCESS`，表示 pending 创建成功。

**初学者理解版：**  
resultStatus 就是“结果怎么样”。  
成功、失败、取消、没权限，这几种最常用。

---

### 16. requestSummary 应该记录多详细？

**专业术语版：**  
requestSummary 应记录足够还原业务意图的摘要，但不应保存敏感信息或过长原文。建议记录用户输入摘要、目标对象、关键参数，并对长文本进行截断，例如限制在 200 到 500 字符。

**初学者理解版：**  
requestSummary 不需要把所有内容都存进去，只要能看懂用户想干什么。  
比如记录“用户请求将工单 #3 状态改为 DONE”，就够了。

---

### 17. resultSummary 应该记录多详细？

**专业术语版：**  
resultSummary 应记录执行结果的业务摘要，例如“成功创建 pending_action”“用户取消操作”“权限不足：USER 不能修改工单状态”“Python 服务调用失败”。它应便于排查问题，但不应包含堆栈、token、密码或完整大模型输出。

**初学者理解版：**  
resultSummary 就是结果说明。  
不要存一大堆报错堆栈，只要写清楚“成功了”“失败原因是什么”即可。

---

### 18. 为什么不能记录 token？

**专业术语版：**  
token 是认证凭证，记录到日志会造成凭证泄露风险。攻击者一旦获取日志中的 JWT 或 refresh token，可能伪造用户身份访问系统。日志通常被多人查看或长期保存，因此不能写入敏感认证信息。

**初学者理解版：**  
token 就像临时钥匙。  
如果把钥匙写进日志，别人看到日志就可能冒充用户登录。

---

### 19. 为什么不能记录 API key？

**专业术语版：**  
API key 是第三方服务或模型服务的访问凭证。泄露后可能造成额度消耗、数据泄露、服务滥用和经济损失。审计日志只能记录“调用了哪个服务”，不能记录密钥本身。

**初学者理解版：**  
API key 就像调用 AI 服务的密码。  
如果写进日志，别人拿到后可能用你的账号调用模型，产生费用。

---

### 20. 如果日志表字段不够，应该先用 extraJson 还是立刻改表？

**专业术语版：**  
如果字段只是用于扩展审计上下文，且当前阶段还在快速迭代，优先使用 extraJson 以降低数据库迁移成本。如果字段将长期作为高频查询条件，例如 conversationId、actionType、resultStatus，则建议后续升级为独立列并建立索引。

**初学者理解版：**  
刚开始不要一缺字段就改表。  
可以先把一些扩展信息放到 extraJson 里。等你发现某个字段经常要筛选，比如 conversationId，再考虑单独加列。

---

## 三、项目展示问题

### 21. AI 查询工单应该怎么记录？

**专业术语版：**  
AI 查询工单应记录为只读操作，推荐：

- operationSource：`AI`
- actionType：`AI_QUERY_TICKET`
- targetType：`TICKET`
- targetId：工单 ID
- resultStatus：`SUCCESS` / `FAILED` / `FORBIDDEN`
- requestSummary：用户查询意图摘要
- resultSummary：查询结果摘要

查询操作通常不需要 pending_action。

**初学者理解版：**  
AI 查工单虽然不改数据，但也要记下来。  
比如记录“staff01 使用 AI 查询了 3 号工单”。

---

### 22. AI 创建工单 pending 应该怎么记录？

**专业术语版：**  
AI 创建工单属于写操作，应先创建 pending_action，并记录：

- actionType：`AI_CREATE_TICKET_PENDING`
- resultStatus：`SUCCESS`
- targetType：`TICKET`
- targetId：创建前可能为空，确认创建后补充或另记 confirmed 日志
- requestSummary：拟创建工单的标题、分类、优先级摘要
- conversationId：当前 AI 会话 ID

**初学者理解版：**  
AI 准备创建工单时，不能直接创建，要先记一条“待确认”。  
确认前它只是“准备创建”，不是“已经创建”。

---

### 23. AI 修改状态确认应该怎么记录？

**专业术语版：**  
用户确认 pending_action 后，系统执行状态变更，并记录：

- actionType：`AI_UPDATE_STATUS_CONFIRMED`
- resultStatus：`SUCCESS`
- targetType：`TICKET`
- targetId：工单 ID
- requestSummary：原状态与目标状态
- resultSummary：状态变更成功，例如 `PENDING -> DONE`

**初学者理解版：**  
用户确认后，AI 修改状态才真正生效。  
日志里要写清楚：哪个工单，从什么状态改成什么状态。

---

### 24. AI 回复建议应该怎么记录？

**专业术语版：**  
AI 回复建议属于辅助生成行为，可记录为：

- actionType：`AI_REPLY_SUGGESTION`
- operationSource：`AI`
- resultStatus：`SUCCESS` / `FAILED`
- targetType：`TICKET`
- targetId：工单 ID
- requestSummary：生成建议的上下文摘要
- resultSummary：建议摘要、confidence、riskFlags 摘要

AI 回复建议本身不应直接成为正式回复。

**初学者理解版：**  
AI 生成建议也要记。  
但这条日志不表示回复已经发给用户，只表示 AI 提供了一个建议。

---

### 25. AI 回复采纳应该怎么记录？

**专业术语版：**  
AI 回复被 STAFF 确认并保存为正式回复时，应记录：

- actionType：`AI_REPLY_CONFIRMED`
- resultStatus：`SUCCESS`
- targetType：`REPLY` 或 `TICKET`
- targetId：回复 ID 或工单 ID
- resultSummary：AI 建议被采纳并保存为正式回复

如果系统支持编辑后采纳，需要记录“采纳的是编辑后的内容摘要”。

**初学者理解版：**  
AI 建议被 staff 点确认并保存后，才算“采纳”。  
日志要记录：这条 AI 建议最后真的变成正式回复了。

---

### 26. 如何统计 AI 建议采纳率？

**专业术语版：**  
AI 建议采纳率可以通过 operation_log 统计：

`AI 建议采纳率 = AI_REPLY_CONFIRMED 次数 / AI_REPLY_SUGGESTION 次数`

如果分母为 0，应返回 0，避免除零异常。统计口径必须稳定，建议统一基于 actionType 统计。

**初学者理解版：**  
AI 建议采纳率就是：AI 给了多少次建议，其中多少次被 staff 采用。  
比如 AI 给了 10 次建议，staff 采纳 6 次，采纳率就是 60%。

---

### 27. 如何判断 AI 操作是否成功？

**专业术语版：**  
不能只根据 AI 是否返回文本判断成功，而应根据业务执行结果判断。例如查询是否成功返回数据、pending_action 是否创建成功、确认后数据库是否完成变更、回复是否持久化成功。最终以 Java 后端业务状态和 resultStatus 为准。

**初学者理解版：**  
AI 说“成功”不一定真的成功。  
要看数据库有没有变、接口有没有返回成功、日志有没有记录 SUCCESS。

---

### 28. 日志页应该支持哪些筛选条件？

**专业术语版：**  
日志页建议支持：

- operationSource：MANUAL / AI；
- actionType：具体操作类型；
- resultStatus：SUCCESS / FAILED / CANCELLED / FORBIDDEN；
- conversationId；
- userId 或 username；
- targetType / targetId；
- 时间范围；
- 分页和排序。

这些筛选条件可以帮助管理员快速定位问题链路。

**初学者理解版：**  
日志多了以后不能一页页翻。  
应该能按 AI 操作、失败记录、某个用户、某个 conversationId 快速筛选。

---

### 29. ADMIN 为什么能看 AI 日志？

**专业术语版：**  
ADMIN 具备系统级管理和审计权限，需要查看 AI 操作日志来进行安全审计、故障排查、权限检查和业务监管。AI 日志可能包含用户行为摘要和系统决策过程，因此只应开放给管理员。

**初学者理解版：**  
管理员负责看系统有没有乱操作，所以能看 AI 日志。  
如果 AI 误操作，管理员需要知道发生了什么。

---

### 30. USER 为什么不能看 AI 日志？

**专业术语版：**  
AI 操作日志可能包含其他用户、员工、工单、权限判断和系统内部执行信息。USER 查看全局 AI 日志会造成越权访问和隐私泄露。普通用户只能查看与自己工单相关的可展示业务信息，而不是系统审计日志。

**初学者理解版：**  
普通用户不能看系统后台记录。  
否则可能看到别人的工单、员工操作、系统内部信息。

---

# 6.5 Dashboard 与数据统计

## 一、基础问题

### 1. Dashboard 是什么？

**专业术语版：**  
Dashboard 是面向管理者或运营者的可视化数据总览页面，用于集中展示关键业务指标、系统状态和运营趋势。它通常由统计卡片、图表、列表和筛选条件组成。

在你的项目中，管理员 Dashboard 用于展示工单总数、状态分布、优先级数量、AI 建议次数和 AI 采纳率。

**初学者理解版：**  
Dashboard 就是后台首页的数据看板。  
管理员一进来就能看到系统现在有多少工单、处理了多少、AI 用得怎么样。

---

### 2. 为什么 Dashboard 适合项目展示？

**专业术语版：**  
Dashboard 能直观体现系统的业务闭环、数据沉淀和管理能力。相比单纯 CRUD 页面，Dashboard 更能展示项目的产品化程度、数据统计能力和后台管理价值。

**初学者理解版：**  
做 Dashboard 很适合演示，因为别人一看就知道你的系统不是玩具项目。  
它能展示真实数据和管理能力。

---

### 3. 什么是统计卡片？

**专业术语版：**  
统计卡片是一种轻量级数据展示组件，用于展示单个关键指标，如总数、增长量、比例、状态计数等。它通常包含标题、数值、单位和简单说明。

**初学者理解版：**  
统计卡片就是一个小方块，上面写着一个数字。  
比如“总工单：128”“AI 采纳率：62.9%”。

---

### 4. 为什么第一版不建议做复杂图表？

**专业术语版：**  
第一版 Dashboard 的目标是快速验证数据链路和权限控制。复杂图表会引入额外依赖、样式维护成本和数据结构复杂度。对于初期项目，数字卡片足以完成演示和验收。

**初学者理解版：**  
先别急着画折线图、饼图。  
你现在最重要的是把真实数据查出来并显示正确。复杂图表可以后面再加。

---

### 5. 什么数据适合放在 ADMIN 首页？

**专业术语版：**  
ADMIN 首页适合放全局运营指标和管理指标，例如工单总数、待处理数量、处理中数量、已完成数量、已关闭数量、高优先级数量、AI 建议次数、AI 采纳次数和 AI 采纳率。

**初学者理解版：**  
管理员首页应该放“系统整体情况”。  
比如有多少工单没处理、AI 建议用了多少次、采纳了多少次。

---

## 二、后端统计问题

### 6. 为什么统计数据应该来自真实数据库？

**专业术语版：**  
Dashboard 是系统运行状态的真实反映，统计数据必须来自持久化数据库或可信数据源。使用 mock 数据会导致展示结果与业务状态不一致，无法用于管理决策和验收。

**初学者理解版：**  
Dashboard 不能写假数字。  
如果数据库有 10 个工单，页面却写死显示 128，那就没有意义。

---

### 7. 什么是 SQL 聚合查询？

**专业术语版：**  
SQL 聚合查询是使用 `COUNT`、`SUM`、`AVG`、`MAX`、`MIN` 等聚合函数对数据库记录进行统计计算的查询方式。例如统计每种状态的工单数量。

**初学者理解版：**  
SQL 聚合就是让数据库帮你数数。  
比如直接问数据库：“每种状态的工单各有多少个？”

---

### 8. `GROUP BY` 有什么用？

**专业术语版：**  
`GROUP BY` 用于按照指定字段对记录分组，并对每个分组执行聚合计算。例如：

```sql
SELECT status, COUNT(*)
FROM ticket
GROUP BY status;
```

这会统计每个状态对应的工单数量。

**初学者理解版：**  
`GROUP BY` 就是“按某个字段分类统计”。  
比如按状态分组，就能知道 PENDING 有多少、DONE 有多少。

---

### 9. 为什么不要查出所有数据再在 Java 里循环统计？

**专业术语版：**  
将所有数据查询到 Java 内存中再统计会增加数据库传输成本、内存占用和响应时间。当数据量变大时容易造成性能问题。聚合统计应该下推到数据库层，由数据库执行更高效的分组和计数。

**初学者理解版：**  
不要把所有工单都拿出来再自己数。  
让数据库直接告诉你结果更快、更省资源。

---

### 10. 统计接口为什么应该只读？

**专业术语版：**  
Dashboard 统计接口用于查询业务指标，不应产生数据变更。只读接口能降低副作用风险，便于缓存、权限控制和接口幂等性保证。

**初学者理解版：**  
看统计数据不应该改变系统。  
打开 Dashboard 只是查看，不应该改工单、改状态或写新数据。

---

### 11. Dashboard 接口为什么只能 ADMIN 访问？

**专业术语版：**  
Dashboard 展示的是全局系统数据，可能包含所有用户、员工和工单的统计信息。普通 USER 或 STAFF 访问会造成越权和数据泄露，因此应限制为 ADMIN 权限。

**初学者理解版：**  
Dashboard 是管理员看的全局后台。  
普通用户不应该知道整个系统有多少工单、哪些工单高优先级。

---

### 12. 没有数据时应该返回 null 还是 0？

**专业术语版：**  
统计接口应返回 0，而不是 null。0 表示统计结果为空但语义明确；null 容易导致前端计算、格式化或渲染异常。尤其采纳率计算时，分母为 0 应返回 0。

**初学者理解版：**  
没有数据就显示 0。  
不要返回 null，否则前端可能报错或显示空白。

---

### 13. 平均处理时间怎么计算？

**专业术语版：**  
平均处理时间通常计算为已完成或已关闭工单的处理耗时平均值：

`AVG(resolved_at - created_at)` 或 `AVG(closed_at - created_at)`

如果系统没有 resolvedAt 或 closedAt 字段，可以暂时不做该指标，或在后续补充状态流转时间记录。

**初学者理解版：**  
平均处理时间就是：每个工单从创建到完成用了多久，然后求平均。  
如果你的表里没有“完成时间”，就先别做这个指标。

---

### 14. AI 建议采纳率怎么计算？

**专业术语版：**  
建议使用 operation_log 中的 actionType 统计：

`aiAcceptanceRate = count(AI_REPLY_CONFIRMED) / count(AI_REPLY_SUGGESTION)`

当建议生成次数为 0 时，采纳率返回 0，避免除零异常。

**初学者理解版：**  
AI 给了多少次建议，staff 采纳了多少次。  
采纳次数除以建议次数，就是采纳率。

---

### 15. 统计数据实时计算和缓存有什么区别？

**专业术语版：**  
实时计算是每次请求 Dashboard 时直接查询数据库，数据准确但查询成本较高。缓存是将统计结果保存一段时间，减少数据库压力，但数据可能有短暂延迟。初期项目建议先实时计算，后续数据量变大再考虑缓存。

**初学者理解版：**  
实时计算就是每次都重新查数据库，比较准。  
缓存就是先存一份结果，下次直接拿，比较快但可能不是最新的。

---

## 三、前端展示问题

### 16. 前端如何展示统计卡片？

**专业术语版：**  
前端可以将 Dashboard stats 接口返回的数据映射为卡片配置数组，例如 title、value、unit、description，然后通过统一的 Card 组件渲染。这样便于维护和扩展。

**初学者理解版：**  
前端拿到后端返回的数字后，把每个数字放进一个小卡片里。  
比如一个卡片显示“总工单”，另一个卡片显示“待处理”。

---

### 17. 数字加载中应该怎么显示？

**专业术语版：**  
前端应在请求进行中展示 loading 状态，例如骨架屏、加载图标或占位符。避免用户看到空白页面或误以为数据为 0。

**初学者理解版：**  
数据还没回来时，不要直接空白。  
可以显示“加载中...”或者一个转圈。

---

### 18. 接口失败时应该怎么提示？

**专业术语版：**  
接口失败时应展示非阻塞错误提示，并保留页面结构。可以显示“统计数据加载失败，请稍后重试”，同时记录错误日志。不要让页面白屏。

**初学者理解版：**  
如果后端接口挂了，前端要告诉用户“加载失败”。  
不要整个页面崩掉。

---

### 19. 403 时应该怎么处理？

**专业术语版：**  
403 表示当前用户已认证但无权限访问资源。前端应展示无权限提示，或跳转到无权限页面。对于 Dashboard，USER 和 STAFF 收到 403 时应提示“无权限访问管理员统计”。

**初学者理解版：**  
403 不是没登录，而是登录了但权限不够。  
普通用户访问管理员页面，就应该提示没权限。

---

### 20. 刷新页面后数据为什么不能丢？

**专业术语版：**  
Dashboard 数据应由后端接口从数据库实时获取，而不是仅保存在前端内存状态中。页面刷新后，前端重新请求接口即可恢复数据，保证数据持久性和一致性。

**初学者理解版：**  
刷新页面后，前端内存会清空。  
所以数据必须存在数据库里，刷新时重新向后端获取。

---

# 6.6 部署与工程化

### 1. Docker Compose 在你的项目中负责什么？

**专业术语版：**  
Docker Compose 用于编排多容器应用，统一管理前端、Java 后端、Python AI 服务、MySQL、Redis、Nginx 等组件的构建、启动、网络和环境变量。它让多服务系统可以用一条命令启动。

**初学者理解版：**  
你的项目不是一个程序，而是一组程序。  
Docker Compose 就像总开关，可以一次启动前端、后端、AI 服务、数据库和 Redis。

---

### 2. Nginx 反向代理负责什么？

**专业术语版：**  
Nginx 反向代理负责统一入口、请求转发、静态资源托管和跨服务路由。例如将 `/api` 转发到 Java 后端，将前端静态页面直接返回给浏览器，将不同服务隐藏在统一域名后面。

**初学者理解版：**  
Nginx 就像门卫。  
用户只访问一个地址，Nginx 决定这个请求应该交给前端、Java，还是其他服务。

---

### 3. 为什么部署后不能硬编码 localhost？

**专业术语版：**  
在容器和服务器环境中，localhost 表示当前容器或当前进程自身，而不是宿主机或其他服务。硬编码 localhost 会导致服务间通信失败。部署时应使用容器服务名、环境变量或 Nginx 代理路径。

**初学者理解版：**  
在 Docker 里，每个容器都有自己的 localhost。  
Java 容器里的 localhost 不是 Python 容器，所以不能写死 localhost。

---

### 4. 前端生产环境 API 地址应该怎么配置？

**专业术语版：**  
前端生产环境 API 地址应通过环境变量或构建配置指定，例如 `VITE_API_BASE_URL=/api`，并由 Nginx 将 `/api` 反向代理到 Java 后端。这样可以避免硬编码服务器 IP 和端口。

**初学者理解版：**  
前端不要写死 `http://localhost:8080`。  
生产环境可以统一请求 `/api`，让 Nginx 帮你转发到后端。

---

### 5. Java 的 `application-dev.yml` 和 `application-prod.yml` 应该怎么区分？

**专业术语版：**  
`application-dev.yml` 用于本地开发环境，配置本地数据库、调试日志、本地 AI 服务地址；`application-prod.yml` 用于生产环境，配置容器服务名、生产数据库、较严格日志级别和环境变量注入的密钥。

**初学者理解版：**  
dev 是本地开发用的，prod 是服务器部署用的。  
本地可以连 localhost，服务器要连 Docker 服务名或生产数据库。

---

### 6. 数据库密码为什么不能写死在代码里？

**专业术语版：**  
数据库密码属于敏感配置，写死在代码中会造成源码泄露风险、环境迁移困难和密钥轮换困难。应通过环境变量、配置中心或部署平台 Secret 注入。

**初学者理解版：**  
数据库密码不能放代码里。  
代码上传 GitHub 后，别人可能看到你的数据库密码。

---

### 7. Redis 在 pending_action 中起什么作用？

**专业术语版：**  
Redis 适合存储短期状态数据。pending_action 需要会话隔离、快速读写、自动过期和确认后删除，因此适合存入 Redis，并通过 TTL 控制有效期。

**初学者理解版：**  
pending_action 是“等用户确认的临时操作”。  
Redis 可以临时保存它，过几分钟自动过期，很适合这个场景。

---

### 8. Java、Python、MySQL、Redis 启动顺序有什么影响？

**专业术语版：**  
Java 后端依赖 MySQL 和 Redis，Python 可能依赖 Java API 或配置服务。如果数据库或 Redis 未就绪，Java 启动可能失败；如果 Java 未启动，Python 调用 Java API 会失败。Docker Compose 可以通过 depends_on 和健康检查改善启动顺序，但应用本身也需要重试机制。

**初学者理解版：**  
后端要先能连上数据库和 Redis。  
如果 MySQL 还没启动好，Java 就可能报错；如果 Java 没启动好，Python 调 Java 也会失败。

---

### 9. 后端日志应该怎么查看？

**专业术语版：**  
在 Docker 部署中，可使用 `docker compose logs` 查看服务日志。例如：

```bash
docker compose logs -f java
docker compose logs -f python
docker compose logs -f nginx
```

也可以结合应用日志级别、日志文件挂载和集中式日志系统进行长期管理。

**初学者理解版：**  
服务器上看日志一般用 Docker 命令。  
比如想看 Java 后端出了什么问题，就执行 `docker compose logs -f java`。

---

### 10. 服务器上如何验证 Docker 容器是否正常运行？

**专业术语版：**  
可以通过容器状态、端口监听、健康检查、服务日志和接口连通性验证。常用命令包括：

```bash
docker compose ps
docker compose logs -f
docker ps
curl http://localhost/api/health
```

同时应检查 MySQL、Redis、Java、Python、Nginx 是否全部处于健康状态。

**初学者理解版：**  
你可以先看容器是不是都在运行，再看日志有没有报错，最后用浏览器或 curl 测接口。  
只要前端能打开、登录能成功、AI 能调用，说明核心链路基本正常。

# 总结

这三部分对应你项目下一阶段最重要的三种能力：

1. **AI 操作审计日志**：证明 AI Agent 的行为可追踪、可解释、可控制。
2. **Dashboard 与数据统计**：证明系统有真实业务数据沉淀和管理展示能力。
3. **部署与工程化**：证明项目不是只能本地运行，而是具备可部署、可维护、可验证的工程能力。
