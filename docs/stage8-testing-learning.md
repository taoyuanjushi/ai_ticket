# 第八阶段学习复盘：为什么要补测试、联调和最终验收

本文用于复盘第八阶段改造：在前面已经完成 typed client、结构化意图识别、pending_action、重新鉴权、结构化 AI 回复、防幻觉规则和新增 AI 能力之后，继续补充自动化测试，把这些业务规则固定下来。

可以先记住一个简单类比：

```text
代码 = 系统真正做事的人
测试 = 每次改代码前后都要核对的检查清单
业务规则 = 系统不能随便改变的底线
回归测试 = 防止以后改代码时把旧功能改坏
```

本阶段重点验证三条线：

```text
Java 后端：权限、JWT、状态流转、pending_action、审计日志、缓存安全
Python AI 服务：JavaTicketClient、意图识别、pending_action、LLM JSON、防幻觉
前端：能否正常构建，说明页面代码和类型检查没有被破坏
```

## 1. 单元测试和集成测试有什么区别？

单元测试可以理解为“只测一个小零件”。

例如：

```text
只测试 IntentRecognizer：
输入：“关闭 3 号工单”
期望：识别为 UPDATE_TICKET_STATUS，ticket_id = 3，target_status = CLOSED
```

这个测试不需要真的启动 Java，不需要连接数据库，也不需要调用 LLM。它只关心一个类或一个函数是否按预期工作。

集成测试可以理解为“测试多个零件装在一起后能不能正常配合”。

例如：

```text
前端带 token 调 Java /ai/chat
Java 把 Authorization 转发给 Python /agent/chat
Python 再通过 JavaTicketClient 调 Java Ticket API
Java 根据当前用户权限返回工单数据
```

这就不是只看一个函数了，而是看多个模块之间的数据、权限、token、错误处理能不能串起来。

简单对比：

| 类型 | 重点 | 优点 | 缺点 |
| --- | --- | --- | --- |
| 单元测试 | 一个函数、一个类、一个服务 | 快、稳定、容易定位问题 | 不能证明完整链路一定可用 |
| 集成测试 | 多个模块一起工作 | 更接近真实使用 | 慢一些，依赖环境更多 |

本项目里两种测试都需要：

```text
单元测试：保证 IntentRecognizer、JavaTicketClient、grounding 这些小模块正确。
集成测试：保证登录、权限、AI 调用、pending_action、缓存安全这些链路正确。
```

## 2. 为什么权限逻辑必须写测试？

权限逻辑决定“谁能看什么、谁能改什么”。

工单系统里，权限错误不是普通 bug，而是安全问题。

例如：

```text
tom 是普通用户，只能看自己的工单。
alice 也是普通用户，只能看自己的工单。
staff 可以处理工单。
admin 可以看操作日志。
```

如果权限代码写错，可能出现：

- 普通用户看到别人的工单。
- 普通用户修改了不该修改的状态。
- 普通用户看到后台操作日志。
- AI 绕过 Java 权限，拿到用户本来不能看的工单内容。

这些问题不一定每次手工测试都能发现，因为很多权限 bug 只在特定角色、特定数据、特定接口组合下出现。

所以权限必须写成自动化测试。

本阶段 Java 测试里重点覆盖：

```text
普通用户不能看他人工单
普通用户不能修改状态
普通用户不能看操作日志
STAFF / ADMIN 按权限访问
缓存命中后仍然不能绕过权限
```

权限测试的价值是：以后有人重构 `TicketService`、`PermissionUtil`、`JwtInterceptor` 或缓存逻辑时，只要不小心放松了权限，测试就会失败。

## 3. 为什么缓存安全也要测试？

缓存是为了加速，但缓存不能绕过权限。

可以把缓存理解为系统旁边的一本“临时笔记本”：

```text
第一次查工单详情：从数据库查，顺便写入缓存。
第二次查同一个工单：如果缓存有，就直接从缓存读。
```

问题在于：缓存通常按 key 取数据，如果 key 设计或读取逻辑不严谨，就可能把用户 A 能看的数据给用户 B。

错误示例：

```text
用户 A 有权限看 ticketId = 10。
系统把 ticket:detail:10 写入缓存。

用户 B 没有权限看 ticketId = 10。
用户 B 请求 ticketId = 10。
系统发现缓存里有 ticket:detail:10，直接返回。
```

这样就出问题了：数据库权限判断被缓存绕过去了。

正确做法是：

```text
即使命中缓存，也要重新检查当前用户是否有权限访问这个工单。
```

本阶段补缓存安全测试，就是为了验证：

```text
用户 A 请求自己的工单详情并写入缓存。
用户 B 请求同一个 ticketId。
即使命中缓存，用户 B 也不能看到用户 A 的工单详情。
普通用户不能命中管理员或其他用户权限范围内的缓存结果。
```

缓存越靠近底层，越容易被人当成“性能优化”随手改动。测试能提醒后续开发者：性能可以优化，但权限不能省。

## 4. 如何测试 pending_action 的幂等性？

幂等可以理解为：

```text
同一个操作执行一次和执行多次，最终结果应该一样。
```

pending_action 场景里，最典型的问题是“重复确认”。

例如用户创建工单：

```text
用户：帮我创建一个工单
系统：已生成待确认操作，是否确认？
用户：确认
系统：真正创建工单
```

如果用户重复点击确认按钮，或者网络重试导致确认请求发送了两次，错误实现可能会变成：

```text
第一次确认：创建 1 张工单
第二次确认：又创建 1 张一样的工单
```

这就不是幂等。

正确结果应该是：

```text
第一次确认：执行业务动作，把 pending_action 从 PENDING 改成 CONFIRMED。
第二次确认：发现它已经不是 PENDING，不再执行业务动作。
```

测试 pending_action 幂等性，一般要验证这些点：

1. 创建 pending_action 后，状态是 `PENDING`。
2. 第一次确认会执行业务动作。
3. 第一次确认后，状态变成 `CONFIRMED`。
4. 第二次确认不会再次执行业务动作。
5. 取消后的 pending_action 不能再确认。
6. 过期后的 pending_action 不能再确认。
7. pending_action 里不能保存 token。

在测试代码里，通常会用 mock 统计业务方法被调用几次。

例如：

```text
确认两次 create_ticket pending_action
期望：TicketService.createTicket 只被调用 1 次
```

这就是用测试证明“重复确认不会重复创建”。

## 5. 如何测试 LLM JSON 输出失败场景？

LLM 不像普通函数，它有时候会输出不符合要求的内容。

我们要求 AI 回复建议必须返回 JSON：

```json
{
  "suggestion": "...",
  "confidence": 0.8,
  "reason": "...",
  "risk_flags": []
}
```

但模型可能会输出：

```text
建议你先安抚用户，然后排查日志。
```

这不是 JSON。

也可能输出：

```json
{
  "suggestion": "请稍后重试",
  "confidence": "很高"
}
```

这虽然像 JSON，但字段类型不对，而且缺少 `reason`、`risk_flags`。

测试 LLM JSON 输出失败，一般不用真的调用外部大模型，而是用假的 LLMClient 模拟不同返回：

```text
场景 1：第一次输出非法 JSON，修复后输出合法 JSON。
期望：服务能修复一次，并返回 ReplySuggestionResult。

场景 2：第一次输出非法 JSON，修复后还是非法。
期望：返回“AI 回复建议生成失败，请稍后重试或手动填写。”

场景 3：输出 JSON 字段缺失或类型错误。
期望：Pydantic 校验失败，然后走修复或降级逻辑。
```

这样测试的好处是：

- 不依赖真实 LLM 网络调用。
- 不花 token 成本。
- 结果稳定，不会今天通过明天失败。
- 能专门覆盖最容易被忽略的异常路径。

在本项目里，LLM JSON 输出失败要满足一个原则：

```text
模型输出不可靠时，系统不能把坏数据继续传下去，更不能自动写数据库。
```

## 6. 为什么测试是防止后续重构破坏业务规则的关键？

重构的目标是让代码更清晰、更好维护，但重构也有风险：表面看起来只是整理代码，实际上可能把业务规则改坏。

例如以后有人想优化代码：

```text
把权限判断抽到新类里
把缓存逻辑重写一遍
把 pending_action 从 Redis 换成数据库
把正则意图识别换成 LLM 意图识别
把 TicketService 拆成多个 service
```

这些改动本身不一定错，但很容易不小心破坏旧规则：

```text
普通用户突然能看别人工单。
缓存命中后忘记再做权限校验。
重复确认又会重复创建。
LLM 输出非法 JSON 时没有降级。
无权限工单仍然调用 LLM 生成建议。
CLOSED 工单又可以被改回 OPEN。
```

如果没有测试，这些问题可能要到用户真实使用时才暴露。

如果有测试，每次重构后只要运行：

```bash
mvn test
pytest
npm run build
```

就能快速知道核心规则有没有被破坏。

所以测试不是“为了让测试数量好看”，而是为了把业务底线写成可执行的规则。

可以把测试理解成项目里的安全护栏：

```text
权限测试：防止越权。
缓存测试：防止缓存泄露数据。
pending_action 测试：防止重复执行写操作。
LLM JSON 测试：防止 AI 输出脏数据。
防幻觉测试：防止 AI 编造工单事实。
状态流转测试：防止工单状态被乱改。
```

这些测试会在后续重构时持续提醒我们：

```text
代码结构可以变，但业务规则不能变。
```

## 本阶段可以记住的结论

单元测试负责检查小模块是否正确，集成测试负责检查多个模块是否能一起工作。

权限、缓存、pending_action、LLM 输出、防幻觉、状态流转都属于工单系统的核心业务规则，必须写测试。

pending_action 的幂等性测试重点是：确认多次也只能执行一次。

LLM JSON 失败测试重点是：模型输出坏格式时，系统要能修复或降级，不能把坏数据写入业务系统。

测试的真正价值不是“证明现在能跑”，而是“防止以后改代码时把重要规则改坏”。
