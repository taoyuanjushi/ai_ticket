# 第六阶段学习复盘：为什么要给 AI 工单回答增加防幻觉规则

本文用于复盘第六阶段改造：所有工单相关 AI 回答都必须基于 Java 后端 `GET /tickets/{id}/detail` 返回的真实工单详情，不能让模型凭感觉编内容。

可以先记住一个简单类比：

```text
LLM = 很会写话的人
ticket_detail = 真实工单档案
grounding = 要求这个人只能看档案说话
防幻觉规则 = 如果档案里没写，就不能假装知道
```

本阶段新增的核心规则在 Python AI 服务里：

```text
ticket-agent-python/app/services/grounding.py
```

其中 `TicketGroundingService` 负责：

- 判断哪些 AI 能力必须先读取工单详情。
- 生成统一的防幻觉 prompt。
- 检查模型输出里有没有无依据结论。
- 信息不足时返回标准提示。

## 1. 什么是 LLM 幻觉？

LLM 幻觉就是：模型说得很像真的，但内容没有可靠依据。

例如工单详情里只写了：

```text
标题：登录失败
描述：用户输入正确密码后仍提示错误
状态：OPEN
```

但 AI 回答：

```text
日志显示数据库已经恢复，问题已经修复。
```

这就是幻觉。

因为真实工单详情里没有：

- 日志内容。
- 数据库状态。
- 修复结果。
- 用户是否已经确认恢复。

模型可能只是根据常见客服话术“猜”了一段看起来合理的话。但在工单系统里，这种猜测很危险。

对业务系统来说，幻觉不是“小错误”，而是可能导致错误操作、错误承诺和错误判断。

## 2. 什么是 grounding？

`grounding` 可以理解为“把 AI 的回答固定在真实依据上”。

在本项目里，真实依据就是 Java 后端返回的 `ticket_detail`：

```text
Python AI 服务
-> JavaTicketClient.get_ticket_detail(auth_token, ticket_id)
-> Java GET /tickets/{id}/detail
-> 返回真实工单标题、描述、状态、优先级、历史回复
```

然后 AI 只能基于这些内容生成回答。

本阶段的防幻觉 prompt 明确写了：

```text
你只能使用 ticket_detail 中的信息。
不能使用外部知识编造工单不存在的事实。
如果 ticket_detail 中没有足够信息，请明确说“当前工单信息不足，需要补充……”
不能假设用户已经做过某些操作。
不能假设系统日志、报错码、处理结果，除非 ticket_detail 中明确出现。
```

简单说：

```text
没有 ticket_detail，不生成工单判断。
ticket_detail 里没有的信息，不允许当成事实说出来。
```

## 3. 为什么业务 Agent 不能只靠模型自由发挥？

模型自由发挥适合写草稿、润色文字、做开放式头脑风暴。

但业务 Agent 不是聊天玩具，它会影响真实系统和真实用户。

工单场景里，AI 可能参与：

- 生成客服回复建议。
- 总结工单。
- 判断优先级。
- 判断分类。
- 提醒 SLA 风险。
- 发起保存 AI 回复的待确认动作。

这些内容如果乱说，会带来真实后果。

例如：

```text
AI 编造“已经修复”
-> 客服误以为问题处理完
-> 用户实际问题没有解决
-> 工单被错误关闭
```

再例如：

```text
AI 编造“日志显示密码错误”
-> 客服把错误原因告诉用户
-> 用户和技术人员被带偏
-> 排查时间变长
```

所以业务 Agent 必须有边界：

```text
模型负责生成语言。
后端负责提供事实。
规则负责限制模型不能越界。
人负责确认关键写操作。
```

如果只靠模型自由发挥，系统就很难保证准确性、权限安全和可审计性。

## 4. 为什么“我不知道 / 信息不足”比编造更安全？

在业务系统里，不知道不是失败，编造才是失败。

如果 AI 说：

```text
当前工单信息不足，需要补充问题现象、操作步骤、错误截图、报错码、日志或处理结果后再继续判断。
```

这虽然不够“聪明”，但它是安全的。

它告诉客服：

- 当前信息不够。
- 不能直接下结论。
- 需要用户或技术人员补充什么。
- 后续可以继续排查。

如果 AI 编造：

```text
日志显示问题已经修复。
```

看起来更像答案，但风险很高：

- 工单里没有日志，AI 却假装看过日志。
- 工单里没有处理结果，AI 却假装已经修复。
- 客服可能直接把错误内容发给用户。
- 后续审计时很难解释这个结论从哪里来。

所以本阶段的原则是：

```text
有依据，就回答。
没依据，就说信息不足。
不确定，就要求补充信息。
```

这比“自信地编一个答案”更适合真实业务系统。

## 5. 如何用后端 API 结果约束模型回答？

本项目用了三层约束。

第一层：先调用 Java 后端拿真实数据。

```text
TicketAiService
-> JavaTicketClient.get_ticket_detail(auth_token, ticket_id)
-> Java /tickets/{id}/detail
```

这一步确保 AI 不是凭空回答，而是先拿到真实工单上下文。

第二层：把 `ticket_detail` 放进 prompt，并写清楚规则。

```text
ticket_detail JSON:
{
  "id": 1,
  "title": "登录失败",
  "description": "用户输入正确密码后仍提示错误",
  "status": "OPEN",
  "priority": "HIGH",
  "replies": []
}
```

同时告诉模型：

```text
只能使用 ticket_detail 中的信息。
不能编造 ticket_detail 中不存在的事实。
```

第三层：模型输出后再检查。

本阶段 `TicketGroundingService` 会检查高风险表述，例如：

```text
已经修复
已经确认
数据库已恢复
用户已解决
日志显示
监控显示
已定位原因
已完成处理
```

如果 AI 输出了这些话，但 `ticket_detail` 里没有依据，系统会降级为信息不足提示，并加上风险标记：

```json
{
  "suggestion": "当前工单信息不足，需要补充问题现象、操作步骤、错误截图、报错码、日志或处理结果后再继续判断。",
  "confidence": 0.6,
  "reason": "ticket_detail 中缺少支持该结论的明确信息，已降级为补充信息提示。",
  "risk_flags": ["信息不足", "需要人工确认"]
}
```

这样就形成了一条安全链路：

```text
真实后端数据
-> 防幻觉 prompt
-> LLM 生成
-> Pydantic 校验 JSON
-> grounding 后置检查
-> 返回安全结果
```

## 6. 为什么权限失败后不能继续调用 LLM 生成建议？

因为权限失败说明当前用户没有资格查看这个工单。

例如 Java 返回：

```text
403 Forbidden
```

意思是：

```text
你没有权限执行该操作。
```

或者 Java 返回：

```text
404 Not Found
```

在本项目里可能表示：

```text
目标工单不存在，或你无权访问该工单。
```

这时候 Python AI 服务不能继续调用 LLM 生成建议。

原因有三个。

第一，不能绕过权限。

如果用户不能看工单详情，AI 也不能帮用户“猜”这个工单是什么。

第二，不能泄露信息。

就算 LLM 没拿到完整详情，如果系统继续生成回答，也可能通过错误提示、上下文或历史内容泄露一点点信息。

第三，没有真实上下文就容易幻觉。

权限失败后，Python 拿不到可信的 `ticket_detail`。如果这时还调用 LLM，模型只能自由发挥，生成内容很可能是编造的。

所以正确流程是：

```text
调用 Java /tickets/{id}/detail
-> 如果成功，才调用 LLM
-> 如果 401/403/404，直接返回 JavaTicketClient 的统一错误信息
-> 不调用 LLM
```

这也是本阶段测试覆盖的重点：无权限工单不能生成建议。
