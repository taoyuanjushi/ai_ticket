# 第五阶段学习复盘：AI 回复建议为什么要改成 JSON 结构化输出

本文用于复盘第五阶段改造：AI 回复建议不再只返回一段自然语言，而是返回结构化 JSON。

本阶段的核心输出格式是：

```json
{
  "suggestion": "建议客服如何回复用户",
  "confidence": 0.8,
  "reason": "为什么给出这个建议",
  "risk_flags": ["需要人工确认"]
}
```

可以先记住一个简单类比：

```text
自然语言回复 = AI 写了一段话
结构化 JSON = AI 填了一张固定格式的表
Pydantic schema = 检查这张表有没有乱填
```

## 1. 为什么 LLM 输出要结构化？

LLM 默认很擅长写自然语言，但自然语言对程序不够稳定。

比如 AI 可能这样回复：

```text
建议先让用户补充错误截图，我觉得这个建议比较可靠。
```

这段话人能看懂，但程序很难稳定地判断：

- 哪一部分是正式回复建议？
- 置信度是多少？
- 为什么这么建议？
- 有没有风险？
- 前端应该怎么展示？
- 后端能不能保存？

改成 JSON 后，程序可以清楚地取字段：

```json
{
  "suggestion": "建议先请用户补充错误截图和发生时间。",
  "confidence": 0.7,
  "reason": "工单描述较少，历史回复中也缺少具体排查结果。",
  "risk_flags": ["信息不足", "需要人工确认"]
}
```

这样前端可以直接展示 `suggestion`，用 `confidence` 做可信度提示，用 `risk_flags` 做风险标签。

简单说：

```text
自然语言适合给人看。
结构化 JSON 同时适合给人看，也适合给程序处理。
```

## 2. JSON schema / Pydantic 如何约束模型输出？

JSON schema 或 Pydantic 的作用是：定义“合格输出长什么样”。

本阶段新增的 schema 是：

```python
class ReplySuggestionResult(BaseModel):
    suggestion: str
    confidence: float = Field(ge=0.0, le=1.0)
    reason: str
    risk_flags: list[str] = Field(default_factory=list)
```

它表达了几条规则：

- `suggestion` 必须是字符串。
- `confidence` 必须是数字，并且在 `0.0` 到 `1.0` 之间。
- `reason` 必须是字符串。
- `risk_flags` 必须是字符串列表，不传时默认为空列表。

如果 LLM 返回：

```json
{
  "suggestion": "请用户补充截图",
  "confidence": 1.5,
  "reason": "信息不足",
  "risk_flags": []
}
```

Pydantic 会拒绝它，因为 `confidence=1.5` 超出了范围。

如果 LLM 返回：

```json
{
  "suggestion": "请用户补充截图"
}
```

Pydantic 也会拒绝它，因为缺少必需的 `confidence` 和 `reason`。

所以 Pydantic 就像入口检查员：AI 可以生成内容，但内容必须先通过检查，系统才继续使用。

## 3. suggestion、confidence、reason、risk_flags 分别解决什么问题？

### suggestion

`suggestion` 是真正给客服看的回复建议。

例如：

```text
建议先请用户补充错误截图、发生时间和具体操作步骤，以便继续排查登录失败原因。
```

后续如果用户确认保存 AI 回复，真正写入 `ticket_reply.content` 的也只应该是 `suggestion`。

### confidence

`confidence` 表示 AI 对这条建议有多大把握。

范围是：

```text
0.0 = 非常没把握
1.0 = 非常有把握
```

它不是绝对真理，只是一个提示。

如果工单信息完整，有标题、描述、状态、优先级、历史回复，置信度可以高一些。

如果工单只有标题，描述很少，也没有历史回复，置信度就应该低一些。

### reason

`reason` 解释 AI 为什么给出这个建议。

它解决的是“可解释性”问题。客服看到建议后，不只是知道“该怎么回复”，还知道“为什么这么回复”。

例如：

```text
工单描述中只说明登录失败，但缺少错误截图和发生时间，因此建议先收集更多排查信息。
```

### risk_flags

`risk_flags` 是风险标签，用来提醒客服注意。

本阶段建议的风险值包括：

```text
信息不足
需要人工确认
涉及权限问题
涉及敏感信息
可能需要升级处理
SLA风险
```

例如：

```json
{
  "risk_flags": ["信息不足", "需要人工确认"]
}
```

它的作用是让系统和客服都能快速知道：这条建议不能无脑使用，需要先看风险。

## 4. 为什么 AI 回复建议不能直接保存？

因为“生成建议”和“保存回复”是两个不同动作。

生成建议只是读操作：

```text
读取工单详情 -> 调 LLM -> 返回建议
```

保存回复是写操作：

```text
把 AI 生成内容写入数据库
```

写操作会改变系统数据，所以必须更谨慎。

AI 可能出现这些问题：

- 理解错工单内容。
- 根据不足的信息写出过度承诺。
- 编造不存在的处理结论。
- 没注意到敏感信息或权限风险。
- 语气不适合直接发给用户。

所以本阶段规则是：

```text
用户只是要“生成建议”：只返回 JSON，不写数据库。
用户要“保存/采纳建议”：创建 pending_action，等待人工确认。
确认后：只保存 suggestion 作为 AI 回复内容。
```

这就是 Human-in-the-loop：AI 先给建议，人确认后再写入系统。

## 5. 什么情况下应该返回“信息不足”？

当 `ticket_detail` 里的信息不足以支撑明确回复时，就应该标记“信息不足”。

常见情况包括：

- 工单只有标题，没有描述。
- 描述太短，例如只有“登录不了”。
- 没有历史回复。
- 没有错误截图、错误码、发生时间、操作步骤等排查信息。
- 用户问题很模糊，无法判断原因。
- 工单状态和已有回复无法说明处理进度。

例如工单详情只有：

```json
{
  "title": "登录失败",
  "description": "",
  "status": "OPEN",
  "priority": "HIGH",
  "replies": []
}
```

AI 不应该编造“已经排查到密码错误”或“已经修复”。

更合适的输出是：

```json
{
  "suggestion": "当前工单信息不足，请先请用户补充错误截图、发生时间和具体操作步骤，再继续排查。",
  "confidence": 0.5,
  "reason": "工单缺少具体描述和历史回复，无法判断登录失败原因。",
  "risk_flags": ["信息不足", "需要人工确认"]
}
```

本阶段服务层也做了兜底：如果工单描述很少、也没有历史回复，即使 LLM 没主动加“信息不足”，系统也会自动补上这个风险标记，并降低 `confidence`。

## 6. 如何处理 LLM 输出非法 JSON？

LLM 虽然被 prompt 要求输出 JSON，但它仍可能返回不标准内容。

例如：

```text
这是建议：
```json
{
  "suggestion": "请用户补充截图",
  "confidence": 0.7,
  "reason": "信息不足",
  "risk_flags": ["信息不足"]
}
```
```

这对人来说很清楚，但对 JSON 解析器来说，外面的“这是建议”和 Markdown 代码块会导致解析失败。

本阶段处理流程是：

```text
1. 先直接用 Pydantic 解析 LLM 输出。
2. 如果失败，尝试修复一次。
3. 修复方式：去掉 Markdown 包裹，截取第一个 { 到最后一个 } 之间的内容。
4. 再用 Pydantic 校验一次。
5. 如果仍然失败，返回可读错误。
```

最终失败时返回：

```text
AI 回复建议生成失败，请稍后重试或手动填写。
```

这样做的好处是：

- 轻微格式问题可以自动恢复。
- 严重格式问题不会被系统强行使用。
- 用户能看到清晰错误，而不是一堆技术异常。
- 系统不会把不可信内容保存进数据库。

```text
LLM 输出要结构化，因为程序需要稳定读取字段。
Pydantic 用来检查 AI 输出是否符合格式。
suggestion 是回复内容，confidence 是把握程度，reason 是解释，risk_flags 是风险提醒。
AI 生成建议只是读操作，保存建议是写操作，必须走 pending_action 确认。
信息不足时不能编造，要标记“信息不足”并说明需要补充什么。

```
