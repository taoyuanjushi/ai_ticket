| 学习问题                   | 你要掌握的答案                                                                               |
| ---------------------- | ------------------------------------------------------------------------------------- |
| React 状态管理是什么？         | 页面上的数据变化都属于状态，例如消息列表、输入框内容、loading、错误、当前 conversationId。状态变化后，React 会重新渲染页面。          |
| 为什么 AI 返回要分类展示？        | 因为不同返回类型需要不同 UI 和交互。普通回答只展示文本，pending 需要 Confirm / Cancel，错误需要重试，JSON 结果需要结构化卡片。      |
| conversationId 为什么重要？  | 它标识一次 AI 会话。创建 pending_action 后，Confirm / Cancel 必须带同一个 conversationId，后端才知道要确认哪一个操作。 |
| 为什么前端不能直接判断权限？         | 前端可以根据结果隐藏按钮或展示提示，但真正权限必须由 Java 后端判断，因为前端代码和请求都可以被用户篡改。                               |
| 为什么 pending 操作要卡片化？    | pending 是一个即将改变数据库的操作，用户需要看清楚操作类型和内容后再确认，避免误操作。                                       |
| 为什么 JSON 结果要卡片化？       | AI 的结构化输出包含不同字段，例如 suggestion、confidence、reason、risk_flags。卡片化能让用户更清楚地理解 AI 结果。       |
| loading 状态有什么用？        | loading 告诉用户系统正在处理请求，避免用户重复点击发送或确认，减少重复请求。                                            |
| retry 重试为什么重要？         | 网络失败、Java 服务异常、Python 服务异常都可能导致请求失败。保留重试能提升体验，也方便调试。                                  |
| 为什么前端不能把 token 放 body？ | token 应该统一放在 Authorization header，由 http 拦截器管理。放 body 容易混乱，也不符合常规认证设计。                |
| 为什么前端不能直连 Python？      | Java 是认证、权限和业务规则入口。前端直连 Python 会绕过 Java 权限，造成越权查询或危险写操作风险。                            |

| 学习问题                               | 参考答案                                                                      |
| ---------------------------------- | ------------------------------------------------------------------------- |
| Entity、DTO、VO 有什么区别？               | Entity 对应数据库表结构，通常和数据库字段接近；DTO 用于接口请求或服务间传输；VO 用于返回给前端展示，可以隐藏敏感字段或组合多个字段。 |
| 新增字段为什么要迁移 SQL？                    | Java 实体新增字段只是代码变化，数据库表也必须新增对应列，否则运行时查询或写入会报错。迁移 SQL 用来同步数据库结构。            |
| 为什么不能随便改表？                         | 改表会影响已有数据、接口返回、缓存、测试、部署、前端类型和 Python DTO。如果没有迁移和兼容处理，线上或 Docker 环境会直接出错。  |
| 为什么 category 先用 String，而不是枚举？      | 初期分类体系不稳定，用 String 更灵活。等分类固定后，再升级为枚举或分类表更合适。                              |
| assignedTo 为什么保存用户 ID，而不是用户名？      | 用户名可能会变，ID 更稳定。保存用户 ID 可以通过关联 User 表获取最新用户信息。                             |
| 为什么普通用户不能修改 category 和 assignedTo？ | 分类和负责人属于内部处理字段，普通用户修改会破坏工单处理流程，应由 STAFF 或 ADMIN 控制。                       |
| 为什么 Service 层也要校验权限？               | Controller、AI、测试或其他入口都可能调用 Service。权限放在 Service 层更安全，避免绕过 Controller。     |
| 为什么修改字段后要清理 Redis 缓存？              | 如果缓存不清理，用户可能继续看到旧的 category 或 assignedTo，甚至在权限变化时出现缓存越权风险。                |
| 操作日志有什么作用？                         | 操作日志记录谁在什么时候修改了什么，方便审计、排查问题和追踪 AI/人工操作。                                   |
| 为什么 SLA 暂时不落库？                     | SLA 需要 deadline、响应时限、解决时限、关闭时间等完整业务规则。当前先做 AI 风险建议更稳，避免过早设计复杂字段。          |

| 多轮对话状态是什么？                                | 多轮对话状态就是系统记住上一轮对话中还没完成的信息，例如用户要创建工单但缺标题，下一轮用户补标题时系统能接上。                          |
| pending_intent 和 pending_action 有什么区别？    | pending_intent 是“参数没问全”，只用于补齐字段，不会写数据库；pending_action 是“参数已完整，等待用户确认”，确认后会执行写操作。 |
| 为什么 pending_intent 也要按 conversationId 隔离？ | 同一个用户可能同时打开多个 AI 会话。conversationId 可以防止 A 会话缺的字段被 B 会话的回复补上。                     |
| 为什么 pending_intent 还要按 userId 隔离？         | 防止不同用户之间串线，避免一个用户补齐或取消另一个用户的操作。                                                  |
| 为什么 pending_intent 不能保存 token？            | token 是敏感登录凭证，不应该持久化到内存、Redis 或数据库中。确认时应该使用当前请求的新 token。                         |
| 为什么字段补齐后还不能直接执行？                          | 创建工单、修改状态属于写操作，会改变数据库。即使字段完整，也必须让用户确认，避免 AI 误解意图后自动执行。                           |
| 为什么缺字段时不能让 LLM 自己猜？                       | 业务系统不能靠猜来写数据库。缺标题、描述、工单 ID 等关键信息时，必须追问用户。                                        |
| 为什么 Python 不能直接写数据库？                      | Python Agent 只负责识别意图和编排工具，真实写操作必须经过 Java Service，这样才能复用权限、状态流转、日志和事务。            |
| 为什么要写 pytest？                             | 多轮对话很容易出状态串线、重复确认、缺字段误执行等问题。pytest 可以把这些规则固定下来，防止后续改坏。                           |

| 为什么要把 AI 能力嵌入业务页面？             | 用户在看工单详情时最需要 AI 辅助，例如生成回复、总结问题、判断优先级。把 AI 能力放进详情页，比让用户去聊天页手动输入更符合真实业务流程。                                          |
| 页面组件如何复用？                      | 把 AI 结果展示拆成独立组件，例如 `AiJsonResultCard`、`PendingActionCard`、`AiErrorCard`。这样 AiPage 和 TicketDetailPage 都能使用同一套展示逻辑。 |
| 什么是业务场景驱动 UI？                  | UI 不是随便堆按钮，而是围绕用户当前任务设计。用户正在处理某个工单时，就应该直接提供与该工单相关的 AI 操作。                                                         |
| 为什么点击按钮本质上还是发送 message？        | 因为后端 AI Agent 当前入口是 `/ai/chat`，它通过自然语言 message 识别意图。按钮只是帮用户自动生成标准 prompt。                                         |
| 为什么工单详情页也需要 conversationId？    | 如果 AI 返回 pending 确认，Confirm / Cancel 必须带同一个 conversationId，后端才能知道要确认哪一个操作。                                        |
| 为什么不能每次点击按钮都新建 conversationId？ | 如果 pending 是在某个 conversationId 下创建的，确认时换了 conversationId，后端就找不到对应的待确认操作。                                          |
| 为什么要复用 AiPage 的结果卡片？           | 同一种 AI 返回结构应该用同一套组件展示，避免两个页面逻辑不一致，也方便后续维护。                                                                        |
| 为什么前端不能直连 Python？              | Java 是认证、权限、日志和业务规则入口。前端直连 Python 会绕过 Java 权限，造成越权风险。                                                             |
| 为什么请求 body 不能传 userId？         | userId 可以被前端篡改，真实用户身份必须由 Java 后端从 JWT 中解析。                                                                        |
| 为什么错误要保留重试按钮？                  | AI 请求可能因为网络、Java、Python 或 LLM 临时失败。重试能提升可用性，也方便开发阶段排查问题。                                                          |

| ------------------------------ | --------------------------------------------------------------- |
| 建议和执行有什么区别？                    | 建议只是 AI 给出的参考，不改变数据库；执行会修改数据库，例如保存回复、修改分类、修改优先级。                |
| 为什么 AI 建议不能自动采纳？               | AI 可能理解错工单内容，也可能给出不合适的建议。业务系统里凡是会写数据库的动作，都应该由人确认后执行。            |
| 操作日志为什么重要？                     | 操作日志能记录是谁、什么时候、对哪个工单做了什么，方便审计、排查问题和追踪 AI 操作。                    |
| 为什么保存 AI 回复也要 pending_action？  | 保存回复会改变工单历史记录，属于写操作。必须让用户确认，避免 AI 自动写入错误内容。                     |
| 为什么分类建议默认不自动保存？                | 分类可能影响统计、筛选和处理流程。AI 可以建议，但是否采纳应由 STAFF 或 ADMIN 确认。              |
| 为什么优先级建议暂时只展示？                 | 修改优先级会影响工单处理顺序，属于更敏感的业务写操作。初期先做建议展示更安全。                         |
| 为什么 Confirm 时还要重新校验权限？         | 用户创建 pending 后，权限可能变化；或者别人尝试确认不属于自己的 pending。Confirm 时必须再次校验。   |
| 什么是幂等？                         | 同一个确认请求重复执行，结果应该和执行一次一样，不能重复保存两条 AI 回复或重复修改数据。                  |
| 为什么 pending_action 不能保存 token？ | token 是敏感登录凭证，不能持久化。Confirm 时应该使用当前请求 header 中的有效 token。        |
| 为什么业务逻辑要放在 Service 层？          | Controller 只是接口入口，Service 才是业务规则中心。放在 Service 层可以防止其他入口绕过权限和规则。 |

| 学习问题                                    | 参考答案                                                                                         |
| --------------------------------------- | -------------------------------------------------------------------------------------------- |
| 单元测试和接口测试区别是什么？                         | 单元测试主要测试一个函数、类或 Service 方法；接口测试通过 HTTP 入口测试完整链路，例如 Controller、认证、权限、Service、数据库。             |
| 为什么权限必须测试？                              | 权限是系统安全边界，重构、缓存、AI 接入都可能意外绕过权限。测试可以固定规则，防止 tom 看见他人工单或修改状态。                                  |
| 幂等测试是什么？                                | 幂等测试验证同一个操作重复执行不会产生重复副作用，例如重复 confirm 不能创建两条工单或保存两条 AI 回复。                                   |
| 为什么 pending_action 要测 confirm 和 cancel？ | pending_action 是写操作安全机制。confirm 应该执行一次，cancel 应该不执行，重复 confirm 和 cancel 后 confirm 都不能造成错误写库。 |
| 为什么 Redis 缓存也要测权限？                      | 缓存容易绕过数据库查询和权限过滤。如果高权限用户把工单缓存了，低权限用户不能因为命中缓存就看到该工单。                                          |
| 为什么接口测试要用真实 token？                      | 真实 token 可以测试 JwtInterceptor、登录上下文和权限解析是否正常。假 token 可能绕过认证链路。                                |
| 为什么不能只测返回状态码？                           | 只测状态码不够，还要确认数据库没有被修改、日志没有误记录、响应体没有泄露敏感数据。                                                    |
| 为什么测试之间要互相隔离？                           | 如果测试依赖执行顺序或上一个测试留下的数据，单独运行会失败，也会导致结果不可信。                                                     |
| 为什么 body.userId 伪造要测试？                  | 前端传来的 userId 可以被篡改。测试可以确认后端只相信 JWT 解析出的 userId。                                              |
| 为什么接口级测试更接近真实项目？                        | 用户真实访问的是 HTTP 接口，不是直接调用 Service。接口测试能覆盖认证、路由、参数校验、权限、数据库和缓存。                                 |

| 学习问题                               | 参考答案                                                                                            |
| ---------------------------------- | ----------------------------------------------------------------------------------------------- |
| 为什么接口层要薄？                          | API 入口只应该负责接收请求、调用 service、返回响应。业务逻辑放在 API 层会让接口文件越来越乱，也不方便测试和复用。                               |
| Client 层负责什么？                      | Client 层封装外部服务调用，例如 JavaTicketClient 负责调用 Java API、处理 HTTP 状态码、超时、非 JSON、连接失败等问题。               |
| Service 层负责什么？                     | Service 层处理业务流程和工具编排，例如根据意图调用对应能力、处理 pending_intent、创建 pending_action、调用 grounding 和 guardrail。 |
| IntentRecognizer 为什么不能调用 Java？     | 它的职责只是识别用户意图和抽取字段。如果它也调 Java，就会混淆“识别”和“执行业务”的边界。                                                |
| AgentToolService 为什么不应该写大量 prompt？ | Prompt 属于具体 AI 能力，例如摘要或分类。放在各自 Capability Service 中更清晰，也方便单独测试。                                 |
| Capability Service 是什么？            | Capability Service 是一个具体 AI 能力模块，例如 ReplySuggestionService 只负责回复建议，SlaRiskService 只负责 SLA 风险。   |
| GroundingService 的作用是什么？           | 它负责先从 Java 获取真实工单数据，确保 AI 只基于有权限的数据生成结果。                                                        |
| GuardrailService 的作用是什么？           | 它负责检查 AI 输出是否可能有幻觉，例如没有依据却说“已经修复”或“日志显示”。                                                       |
| ResponseBuilder 为什么重要？             | 它保证所有返回格式一致，前端只需要解析 `type / message / data / risk_flags`。                                       |
| 为什么分层后更容易测试？                       | 每一层职责单一后，可以单独 mock 依赖并测试，例如只测试 AgentToolService 是否正确分发，不需要真的调用 LLM。                             |
