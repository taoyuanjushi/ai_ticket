# 第七阶段学习复盘：新增 AI 能力为什么要拆分成摘要、建议、检索和风险判断

本文用于复盘第七阶段改造：在权限、人工确认、防幻觉规则稳定后，Python AI 服务新增了五类能力：

```text
工单摘要
优先级建议
分类建议
相似工单检索
SLA 风险提醒
```

可以先记住一个简单类比：

```text
摘要 = 帮你读完工单后做笔记
分类 = 判断这张工单属于哪一类
建议 = 给出下一步参考意见
检索 = 帮你找相似历史工单
风险判断 = 提醒这张工单可能有什么业务风险
```

本阶段新增的能力代码在：

```text
ticket-agent-python/app/services/ai_capabilities/
```

每个能力都有自己的 service，而不是全部写进 `/agent/chat`。

## 1. 摘要、分类、建议、检索、风险判断分别属于什么 AI 任务？

这五类能力虽然都叫“AI 能力”，但它们其实属于不同任务类型。

### 摘要：Summarization

摘要任务的目标是：把一大段信息压缩成更短、更清楚的内容。

在工单系统里，摘要会读取：

- 工单标题。
- 工单描述。
- 当前状态。
- 优先级。
- 分类。
- 历史回复。

然后输出：

```json
{
  "summary": "1号工单《登录失败》当前状态为 OPEN，优先级为 HIGH。",
  "key_points": ["标题：登录失败", "状态：OPEN"],
  "risk_flags": []
}
```

它不是创造新信息，而是把已有信息整理出来。

所以摘要属于：

```text
信息压缩 + 重点提取
```

### 分类：Classification

分类任务的目标是：把一条数据放进某个类别。

例如工单内容是：

```text
用户输入正确密码后仍无法登录
```

AI 可以建议分类为：

```text
账号登录
```

输出格式是：

```json
{
  "suggested_category": "账号登录",
  "confidence": 0.81,
  "reason": "工单标题或描述中出现了与账号登录相关的关键词。",
  "risk_flags": []
}
```

分类属于：

```text
文本分类 + 业务标签判断
```

### 建议：Recommendation / Decision Support

建议任务的目标是：给人一个参考意见，但不替人直接执行。

本阶段的建议类能力主要包括：

- 优先级建议。
- 分类建议。
- 后续也可以扩展为处理建议、分派建议。

例如优先级建议输出：

```json
{
  "suggested_priority": "HIGH",
  "confidence": 0.76,
  "reason": "工单内容包含登录失败、支付、订单、超时、阻塞或严重失败等高影响关键词。",
  "risk_flags": ["需要人工确认"]
}
```

建议属于：

```text
辅助决策
```

注意：辅助决策不是自动决策。AI 给建议，人或业务规则做最终决定。

### 检索：Retrieval / Search

检索任务的目标是：从已有数据里找出相关内容。

相似工单检索就是：

```text
当前工单 -> 提取关键词 -> 调 Java 查询接口 -> 返回相似工单
```

输出格式是：

```json
{
  "similar_tickets": [
    {
      "id": 12,
      "title": "登录接口报错",
      "status": "CLOSED",
      "similarity_reason": "都涉及登录失败"
    }
  ],
  "risk_flags": []
}
```

检索属于：

```text
信息召回 + 结果筛选
```

第一版使用关键词检索，后续可以升级为向量检索。

### 风险判断：Risk Assessment

风险判断的目标是：根据业务字段判断是否可能有风险。

SLA 风险提醒会看：

- 工单状态。
- 优先级。
- 创建时间。
- 截止时间。
- 回复时间。
- 解决期限。

输出格式是：

```json
{
  "sla_risk_level": "MEDIUM",
  "reason": "高优先级工单仍处于 OPEN，可能需要尽快处理",
  "missing_fields": ["resolveDueAt"],
  "risk_flags": ["SLA字段不足"]
}
```

风险判断属于：

```text
业务规则判断 + 风险提示
```

它不是简单聊天，而是要依赖明确的业务字段。

## 2. 为什么建议类能力默认不直接写数据库？

因为“给建议”和“改数据”是两件完全不同的事。

AI 建议只是参考：

```text
这张工单可能是 HIGH 优先级。
这张工单可能属于账号登录分类。
```

但写数据库会改变真实业务数据：

```text
把 priority 改成 HIGH。
把 category 保存为账号登录。
```

一旦写库，就会影响：

- 用户看到的工单状态。
- 客服处理顺序。
- 报表统计。
- SLA 计算。
- 审计日志。

AI 可能判断错，例如：

```text
AI 把普通咨询误判成 HIGH
-> 工单队列被错误排序
-> 真正紧急的问题被挤到后面
```

所以本阶段规则是：

```text
建议类能力默认只返回 JSON，不写数据库。
如果以后要保存建议结果，必须走 pending_action 人工确认。
```

也就是说：

```text
AI 可以建议。
人确认后，系统才允许写库。
```

这和之前的 `create_ticket`、`update_ticket_status`、`save_ai_reply` 一样，写操作必须进入确认流程。

## 3. 关键词检索和向量检索有什么区别？

关键词检索和向量检索都是“找相关内容”，但思路不同。

### 关键词检索

关键词检索是按字面词匹配。

例如当前工单是：

```text
登录失败，接口超时
```

系统提取关键词：

```text
登录
超时
接口
```

然后去 Java 查询接口里搜索包含这些词的工单。

优点：

- 实现简单。
- 容易调试。
- 结果可解释。
- 不需要向量数据库。
- 适合第一版快速上线。

缺点：

- 只认字面相同或相近的词。
- 用户换一种说法可能搜不到。
- 语义理解能力弱。

例如：

```text
“无法登录”
“进不了系统”
```

这两个意思相近，但关键词不完全一样，关键词检索可能漏掉。

### 向量检索

向量检索是先把文本变成一组数字，也就是 embedding。

它不是只看字面词，而是看语义相似度。

例如：

```text
“无法登录”
“进不了系统”
```

虽然字面不同，但语义接近，向量检索更容易找出来。

优点：

- 能理解语义相似。
- 召回能力更强。
- 适合知识库、历史工单、FAQ 检索。

缺点：

- 需要 embedding 模型。
- 通常需要向量数据库。
- 调试比关键词检索复杂。
- 必须额外做权限过滤。

本阶段先用关键词检索，是因为它更适合第一版：

```text
先保证链路正确、安全、可解释。
以后再升级向量检索。
```

## 4. 相似工单检索为什么必须做权限过滤？

因为相似工单里可能包含用户无权查看的信息。

假设系统里有两张工单：

```text
工单 A：普通用户自己的登录问题
工单 B：另一个用户的账号泄露问题
```

它们可能都包含“登录”“账号”等关键词。

如果相似工单检索不做权限过滤，用户 A 查询自己的工单时，可能看到用户 B 的敏感工单。

这会造成数据泄露。

所以正确流程必须是：

```text
先从当前工单提取关键词
-> 调 Java 查询接口
-> Java 根据当前 auth_token 过滤权限
-> Python 只接收当前用户有权访问的工单
-> Python 排除当前工单
-> 返回相似工单
```

本阶段相似工单检索不直接查 MySQL，也不绕过 Java。

原因很简单：

```text
Java 后端掌握权限规则。
Python AI 服务只负责 AI 编排和结果组织。
```

如果以后升级为向量检索，也必须遵守同样原则：

```text
向量召回候选结果
-> 再经过 Java 权限校验
-> 只返回有权限的结果
```

不能因为用了 AI 检索，就绕过业务权限。

## 5. SLA 判断依赖哪些业务字段？

SLA 是 Service Level Agreement，简单理解就是“服务时限要求”。

比如：

```text
高优先级工单必须 2 小时内响应，24 小时内解决。
```

要判断 SLA 风险，通常需要这些字段：

```text
createdAt       工单创建时间
priority        优先级
status          当前状态
lastReplyAt     最近回复时间
deadline        总截止时间
responseDueAt   响应截止时间
resolveDueAt    解决截止时间
```

本阶段代码里也按这些字段检查：

```text
deadline
createdAt
priority
status
lastReplyAt
responseDueAt
resolveDueAt
```

不同字段解决的问题不同。

`createdAt` 用来知道工单从什么时候开始计时。

`priority` 用来判断时限规则可能有多严格。

`status` 用来判断工单是否已经关闭。

`lastReplyAt` 用来判断客服是否已经响应过。

`responseDueAt` 用来判断响应是否快超时。

`resolveDueAt` 用来判断解决是否快超时。

`deadline` 可以作为总截止时间。

如果这些字段缺失，AI 只能做启发式提醒，例如：

```text
高优先级工单仍未关闭，可能需要尽快处理。
```

但不能给出精确倒计时。

## 6. 没有 deadline 时为什么不能给出精确超时判断？

因为精确超时判断必须有明确时间依据。

如果没有 `deadline`、`responseDueAt` 或 `resolveDueAt`，系统其实不知道：

- 什么时候算超时。
- 还剩多少时间。
- 是否已经违反 SLA。
- 不同优先级对应什么时限。

这时如果 AI 说：

```text
还有 2 小时超时。
```

这就是编造。

因为它没有真实字段支撑这个结论。

正确说法应该是：

```text
高优先级工单仍处于 OPEN，可能需要尽快处理。当前缺少 resolveDueAt，无法给出精确超时判断。
```

所以本阶段 SLA 风险提醒会返回：

```json
{
  "sla_risk_level": "MEDIUM",
  "reason": "高优先级工单仍未关闭，可能需要尽快处理。缺少精确 SLA 字段，不能编造截止时间。",
  "missing_fields": ["deadline", "lastReplyAt", "responseDueAt", "resolveDueAt"],
  "risk_flags": ["SLA字段不足"]
}
```

这里的重点是：

```text
可以提醒风险。
不能编造时间。
可以说字段不足。
不能假装字段存在。
```

这也是第六阶段防幻觉规则在第七阶段能力中的继续应用。

```text
摘要是信息压缩，分类是标签判断，建议是辅助决策，检索是找相关工单，风险判断是业务规则提醒。
建议类能力默认不写库，因为 AI 建议可能错，写库必须走 pending_action 确认。
关键词检索看字面词，向量检索看语义相似。

```
